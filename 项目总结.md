# EKA2L1 WebAssembly 移植 — 项目总结

> 整理时间：2026-06-10
> 汇总自五轮调试记录：[cursor.md](./cursor.md) → [claude.md](./claude.md) → [claude2.md](./claude2.md) → [kilo.md](./kilo.md) → [claude3.md](./claude3.md)

---

## 一、项目目标

把 EKA2L1（Symbian OS / S60 模拟器）通过 Emscripten 移植到浏览器（WebAssembly），实现：

1. 浏览器中可启动模拟器（SDL2 + WebGL2 / OpenGL ES 3.0）
2. 可上传并安装 ROM（EKA1）/ ROM+RPKG（EKA2）
3. 可安装 SIS 软件包，列出已装 app
4. 点击 app 可运行、有画面、可交互
5. 数据持久化（IndexedDB / IDBFS），刷新页面后设备与 app 列表恢复

## 二、移植架构

| 层 | 桌面版 | WASM 版 |
|---|---|---|
| 前端入口 | Qt（多线程：OS 线程 + Graphics 线程） | `src/emu/web/src/main.cpp`（单主线程 + RAF 主循环） |
| 窗口/输入 | Qt / SDL | SDL2（Emscripten 内置端口） |
| 图形 | OpenGL（独立 graphics 线程跑 `driver->run()`） | WebGL2（GL 调用只能在主线程） |
| CPU 模拟 | dynarmic JIT | dyncom 解释器（WASM 无法 JIT arm 代码） |
| 文件系统 | 本机 FS | MEMFS + IDBFS（挂载 `/eka2l1`，IndexedDB 持久化） |
| 主循环 | 各线程自由阻塞 | `emscripten_set_main_loop`（RAF），每帧 2ms CPU 预算、最多 8 个 slice |
| 页面 UI | — | `src/emu/web/shell.html`（上传 ROM/RPKG/SIS、app 列表、暂停/继续） |

关键约束：**Emscripten 主线程禁止阻塞**（`Atomics.wait` 不可用），所有 `join()` / `future::wait()` / `condition_variable::wait()` 在主线程上都会死锁或忙等。这是整个移植过程中最大的错误来源。

## 四、本轮新发现（2026-06-10）：渲染与输入链路根本没接

排查"app 启动后黑屏"时发现，**WASM 前端缺了两条核心链路**，这才是黑屏的真正根因（mprotect、launch 命令等都只是前置问题）：

### 1. 渲染链路缺失

EKA2L1 的渲染模型：窗口服务器（winserv）把绘制命令组装成 `command_list`，`submit_command_list()` 推入 `request_queue`；**必须有消费者执行 `dispatch()` 才会发生真正的 GL 调用**。

- 桌面版：独立 graphics 线程跑 `graphics_driver->run()` 阻塞消费队列（[qt/src/thread.cpp:250](src/emu/qt/src/thread.cpp#L250)）
- Android 版：另外还给每个屏幕注册 `add_screen_redraw_callback`，回调里把 `scr->screen_texture` 画到 backbuffer 并 `present()`（[android .../state.cpp:62](src/emu/android/app/src/main/cpp/src/state.cpp#L62)）
- **WASM 版：driver 创建后队列从未被消费，重绘回调也从未注册** → 所有 GL 命令积压，永远黑屏

更严重的连锁：`request_queue::push()` 在队列满 128 条时**阻塞等消费者**（[common/queue.h:177](src/emu/common/include/common/queue.h#L177)）。WASM 上没有消费者，主线程一旦在 push 处阻塞就是永久死锁/忙等——很可能与"第二个 CPU 100% 热点"有关。

### 2. 输入链路缺失

`sdl_web_window` 在 `init_emulator()` 里是局部变量，函数返回即销毁；input hooks（`raw_mouse_event` / `button_pressed` 等）从未挂接；main_loop 里 `SDL_PollEvent` 把除 QUIT 外的所有事件直接丢弃 → **鼠标/触摸/键盘永远到不了 winserv**。

### 修复方案（本轮实施）

1. `request_queue` 增加非阻塞 `try_pop()`；`graphics_driver` 基类增加 `virtual void pump()`，OGL driver 实现为排空队列并 dispatch —— 主线程每帧 RAF 时调用（GL 调用天然回到主线程，符合 WebGL 约束）
2. WASM 下 `max_pending_count_` 调大，杜绝 push 阻塞死锁
3. 仿照 Android `register_draw_callback` 注册屏幕重绘回调：组装"清屏 + 把 screen_texture 等比缩放画到窗口 + present"命令列表提交（回调可能跑在 timer 线程，只组命令不碰 GL，安全）
4. `sdl_web_window` 保活到全局状态，挂接 mouse/key hooks → `winserv->queue_input_from_driver()`；main_loop 改用 `window->poll_events()`

## 五、关键经验（移植 checklist）

1. **主线程不可阻塞**：`join/wait/lock` 全部要审计。生产者-消费者队列若两端都可能在主线程，必须非阻塞化。
2. **`std::locale` / `std::stringstream` / `std::ifstream` 在 Emscripten libc++ 中都可能崩**：locale 数据库不存在；热路径用 C stdio / snprintf。
3. **默认栈太小**：主线程 64KB → 8MB；worker 64KB → 2MB。复杂 C++ 模板调用链轻松超 64KB。
4. **`-fexceptions` 必须全量重编**，只给最终 target 加不够。
5. **WASM 无 mprotect / 无 JIT**：内存保护走 W^X 兼容路径；CPU 只能用解释器（dyncom）。
6. **`-s ASSERTIONS=1` 是最好的诊断工具**：stack cookie 报告比裸的 "memory access out of bounds" 信息量大一个数量级。
7. **性能诊断**：先关 spdlog（WASM 下每条日志 = 同步跨边界 + console DOM 渲染）；用浏览器任务管理器看真实 CPU；Performance trace 样本数 ≠ 调用次数。
8. **热路径 `unordered_map` 要 reserve + 限容**，rehash 在 dlmalloc 多线程加锁下代价巨大。
9. **postRun ≠ main() 已运行**；IDBFS 的 db handle 会过期；浏览器会强缓存大 js 文件。
10. **对照成熟移植端（Android/Qt）逐条核对前端职责清单**：窗口、输入、渲染队列消费、屏幕合成、音频、生命周期回调——漏一条就是"黑屏/没声/没输入"。

## 八、第 6 轮（2026-06-10 下午）：渲染验证、输入接通、崩溃修复、性能优化

第四节补齐的渲染链路经浏览器验证**成功**——非触屏 S60v3 计算器 ROM 画面、按钮、七段数码管、主题背景全部正常渲染，readback 确认 `canvas=360x640 drawbuf=360x640` 像素真实。本轮在此基础上收尾并优化。

### 1. 诊断清理

调通渲染期间在 `main.cpp` 埋的诊断全部移除：红色清屏、蓝色方块、`[present]` / readback（每帧 `glReadPixels` + `EM_ASM` 探测，最耗性能）、`[hb]` / `[hb-threads]` / `[hb-code]` 心跳（持锁遍历线程 + 反汇编 dump）。保留 `s_redraw_cb_count`——它已被 `main_loop` 的 `boot_phase` 自适应 CPU 预算当作功能逻辑使用。

### 2. 输入链路接通（键盘）

| 环节 | 实现 |
|---|---|

### 6. 性能优化（编译参数开到最高）

| 项 | 之前 | 现在 | 效果 |
|---|---|---|---|
| 链接优化 | `-O2` | `-O3` | wasm-opt 更激进 |

## 九、第 7 轮（2026-06-10 晚）：dyncom 缓存灾难修复 + 前端双页重构

### 1. 找到"第二个 CPU 100% 热点"：每个 slice 全清翻译缓存

排查 `thread_scheduler` 时发现致命链条：

```
system_impl::loop()  每个 slice 结尾
  → kern_->reschedule()
    → switch_context(oldt, newt)   ← 即使 oldt == newt 也不短路
      → run_core->load_context(ctx)
        → clear_instruction_cache()   ← 上游 dyncom 原版行为
```

也就是说**每跑约 2 万条指令就把整个翻译缓存清空一次**，所有热代码块永远在重新 decode+翻译。
上游没人发现是因为桌面端全用 dynarmic/12l1r JIT，dyncom 只是冷门 fallback；WASM 恰好只能用它。

**修复**（保持与上游相同的正确性语义，失效点从"每次线程切换"收敛到真正需要的位置）：

| 失效点 | 之前 | 现在 |
|---|---|---|
| 同线程继续 / 同进程线程切换 | 全清 | **保留全部翻译** |
| 进程（地址空间）切换 | 全清（借道 load_context） | `flush_tlb()` 全清（调度器本就只在换进程时调它） |
| 页被 unmap（DLL 卸载/chunk 回收） | 依赖下次线程切换兜底 | `dirty_tlb_page()` 用**代码页位图**过滤：数据页（堆收缩）零开销，命中代码页才全清 |
| 显式 IMB / 内核 reset | 全清 | 不变 |
| 翻译缓冲耗尽 | 仅条目数阈值（assert 在 release 被编译掉，有溢出风险） | 条目数 + **字节水位**（距 125MB 上限留 1MB 余量）双保险 |

### 2. DISPATCH 直接映射 lookaside

每个跳转都要 `unordered_map::find`（哈希+桶链）。在它前面加 8192 项直接映射表
（`{pc, generation, ptr}`，O(1) 代际失效），热循环里基本全命中，省掉哈希查找。

### 3. 其他热路径收敛

- dyncom watchdog 时钟检查 1024 → 8192 条指令一次（每次 `system_clock::now()` 都跨 wasm→JS 边界）
- `system_impl::loop()` 去掉每 slice 两次时间戳诊断；去掉 2 万 tick 上限——watchdog（8ms 墙钟）才是
  响应性边界，slice 加长 = reschedule/锁开销摊薄

### 4. 前端大改：双页架构（参考 j2me 模拟器 myjump 的 index/main 分离）

`shell.html`（单页 1554 行）退役，拆为**游戏库页**与**运行页**，安装存储与游戏执行解耦，便于移植：

```
src/emu/web/pages/
  index.html   游戏库：游戏列表(搜索/avatar)、安装 SIS、安装设备 ROM+RPKG、设置、数据管理
  run.html     运行页：canvas 舞台 + 虚拟键盘(摇杆区+数字区) + 暂停/存档/全屏

  js/index.js  库页逻辑：localStorage 缓存 app 列表(秒开)、安装流程、设置
  js/run.js    运行页逻辑：?uid= 启动、首帧轮询(redraw count)隐藏 overlay、
```
