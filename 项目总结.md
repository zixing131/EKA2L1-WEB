# EKA2L1 WebAssembly 移植 — 项目总结

> 整理时间：2026-06-10
> 汇总自五轮调试记录：[cursor.md](./cursor.md) → [claude.md](./claude.md) → [claude2.md](./claude2.md) → [kilo.md](./kilo.md) → [claude3.md](./claude3.md)

---

## 一、项目目标

把 EKA2L1（Symbian OS / S60 模拟器）通过 Emscripten 移植到浏览器（WebAssembly），实现：

1. 浏览器中可启动模拟器（SDL2 + WebGL2 / OpenGL ES 3.0）
2. 可上传并安装 ROM（EKA1）/ ROM+RPKG（EKA2）
3. 可安装 SIS 软件包，列出已装 app
4. 点击 app 可运行、有画面、可交互
5. 数据持久化（IndexedDB / IDBFS），刷新页面后设备与 app 列表恢复

## 二、移植架构

| 层 | 桌面版 | WASM 版 |
|---|---|---|
| 前端入口 | Qt（多线程：OS 线程 + Graphics 线程） | `src/emu/web/src/main.cpp`（单主线程 + RAF 主循环） |
| 窗口/输入 | Qt / SDL | SDL2（Emscripten 内置端口） |
| 图形 | OpenGL（独立 graphics 线程跑 `driver->run()`） | WebGL2（GL 调用只能在主线程） |
| CPU 模拟 | dynarmic JIT | dyncom 解释器（WASM 无法 JIT arm 代码） |
| 文件系统 | 本机 FS | MEMFS + IDBFS（挂载 `/eka2l1`，IndexedDB 持久化） |
| 主循环 | 各线程自由阻塞 | `emscripten_set_main_loop`（RAF），每帧 2ms CPU 预算、最多 8 个 slice |
| 页面 UI | — | `src/emu/web/shell.html`（上传 ROM/RPKG/SIS、app 列表、暂停/继续） |

关键约束：**Emscripten 主线程禁止阻塞**（`Atomics.wait` 不可用），所有 `join()` / `future::wait()` / `condition_variable::wait()` 在主线程上都会死锁或忙等。这是整个移植过程中最大的错误来源。

## 四、本轮新发现（2026-06-10）：渲染与输入链路根本没接

排查"app 启动后黑屏"时发现，**WASM 前端缺了两条核心链路**，这才是黑屏的真正根因（mprotect、launch 命令等都只是前置问题）：

### 1. 渲染链路缺失

EKA2L1 的渲染模型：窗口服务器（winserv）把绘制命令组装成 `command_list`，`submit_command_list()` 推入 `request_queue`；**必须有消费者执行 `dispatch()` 才会发生真正的 GL 调用**。

- 桌面版：独立 graphics 线程跑 `graphics_driver->run()` 阻塞消费队列（[qt/src/thread.cpp:250](src/emu/qt/src/thread.cpp#L250)）
- Android 版：另外还给每个屏幕注册 `add_screen_redraw_callback`，回调里把 `scr->screen_texture` 画到 backbuffer 并 `present()`（[android .../state.cpp:62](src/emu/android/app/src/main/cpp/src/state.cpp#L62)）
- **WASM 版：driver 创建后队列从未被消费，重绘回调也从未注册** → 所有 GL 命令积压，永远黑屏

更严重的连锁：`request_queue::push()` 在队列满 128 条时**阻塞等消费者**（[common/queue.h:177](src/emu/common/include/common/queue.h#L177)）。WASM 上没有消费者，主线程一旦在 push 处阻塞就是永久死锁/忙等——很可能与"第二个 CPU 100% 热点"有关。

### 2. 输入链路缺失

`sdl_web_window` 在 `init_emulator()` 里是局部变量，函数返回即销毁；input hooks（`raw_mouse_event` / `button_pressed` 等）从未挂接；main_loop 里 `SDL_PollEvent` 把除 QUIT 外的所有事件直接丢弃 → **鼠标/触摸/键盘永远到不了 winserv**。

### 修复方案（本轮实施）

1. `request_queue` 增加非阻塞 `try_pop()`；`graphics_driver` 基类增加 `virtual void pump()`，OGL driver 实现为排空队列并 dispatch —— 主线程每帧 RAF 时调用（GL 调用天然回到主线程，符合 WebGL 约束）
2. WASM 下 `max_pending_count_` 调大，杜绝 push 阻塞死锁
3. 仿照 Android `register_draw_callback` 注册屏幕重绘回调：组装"清屏 + 把 screen_texture 等比缩放画到窗口 + present"命令列表提交（回调可能跑在 timer 线程，只组命令不碰 GL，安全）
4. `sdl_web_window` 保活到全局状态，挂接 mouse/key hooks → `winserv->queue_input_from_driver()`；main_loop 改用 `window->poll_events()`

## 五、关键经验（移植 checklist）

1. **主线程不可阻塞**：`join/wait/lock` 全部要审计。生产者-消费者队列若两端都可能在主线程，必须非阻塞化。
2. **`std::locale` / `std::stringstream` / `std::ifstream` 在 Emscripten libc++ 中都可能崩**：locale 数据库不存在；热路径用 C stdio / snprintf。
3. **默认栈太小**：主线程 64KB → 8MB；worker 64KB → 2MB。复杂 C++ 模板调用链轻松超 64KB。
4. **`-fexceptions` 必须全量重编**，只给最终 target 加不够。
5. **WASM 无 mprotect / 无 JIT**：内存保护走 W^X 兼容路径；CPU 只能用解释器（dyncom）。
6. **`-s ASSERTIONS=1` 是最好的诊断工具**：stack cookie 报告比裸的 "memory access out of bounds" 信息量大一个数量级。
7. **性能诊断**：先关 spdlog（WASM 下每条日志 = 同步跨边界 + console DOM 渲染）；用浏览器任务管理器看真实 CPU；Performance trace 样本数 ≠ 调用次数。
8. **热路径 `unordered_map` 要 reserve + 限容**，rehash 在 dlmalloc 多线程加锁下代价巨大。
9. **postRun ≠ main() 已运行**；IDBFS 的 db handle 会过期；浏览器会强缓存大 js 文件。
10. **对照成熟移植端（Android/Qt）逐条核对前端职责清单**：窗口、输入、渲染队列消费、屏幕合成、音频、生命周期回调——漏一条就是"黑屏/没声/没输入"。

## 八、第 6 轮（2026-06-10 下午）：渲染验证、输入接通、崩溃修复、性能优化

第四节补齐的渲染链路经浏览器验证**成功**——非触屏 S60v3 计算器 ROM 画面、按钮、七段数码管、主题背景全部正常渲染，readback 确认 `canvas=360x640 drawbuf=360x640` 像素真实。本轮在此基础上收尾并优化。

### 1. 诊断清理

调通渲染期间在 `main.cpp` 埋的诊断全部移除：红色清屏、蓝色方块、`[present]` / readback（每帧 `glReadPixels` + `EM_ASM` 探测，最耗性能）、`[hb]` / `[hb-threads]` / `[hb-code]` 心跳（持锁遍历线程 + 反汇编 dump）。保留 `s_redraw_cb_count`——它已被 `main_loop` 的 `boot_phase` 自适应 CPU 预算当作功能逻辑使用。

### 2. 输入链路接通（键盘）

| 环节 | 实现 |

| 项 | 之前 | 现在 | 效果 |

| 链接优化 | `-O2` | `-O3` | wasm-opt 更激进 |

## 九、第 7 轮（2026-06-10 晚）：dyncom 缓存灾难修复 + 前端双页重构

### 1. 找到"第二个 CPU 100% 热点"：每个 slice 全清翻译缓存

排查 `thread_scheduler` 时发现致命链条：

```
system_impl::loop()  每个 slice 结尾
  → kern_->reschedule()
    → switch_context(oldt, newt)   ← 即使 oldt == newt 也不短路
      → run_core->load_context(ctx)
        → clear_instruction_cache()   ← 上游 dyncom 原版行为
```

也就是说**每跑约 2 万条指令就把整个翻译缓存清空一次**，所有热代码块永远在重新 decode+翻译。
上游没人发现是因为桌面端全用 dynarmic/12l1r JIT，dyncom 只是冷门 fallback；WASM 恰好只能用它。

**修复**（保持与上游相同的正确性语义，失效点从"每次线程切换"收敛到真正需要的位置）：

| 失效点 | 之前 | 现在 |
|---|---|---|
| 同线程继续 / 同进程线程切换 | 全清 | **保留全部翻译** |
| 进程（地址空间）切换 | 全清（借道 load_context） | `flush_tlb()` 全清（调度器本就只在换进程时调它） |
| 页被 unmap（DLL 卸载/chunk 回收） | 依赖下次线程切换兜底 | `dirty_tlb_page()` 用**代码页位图**过滤：数据页（堆收缩）零开销，命中代码页才全清 |
| 显式 IMB / 内核 reset | 全清 | 不变 |
| 翻译缓冲耗尽 | 仅条目数阈值（assert 在 release 被编译掉，有溢出风险） | 条目数 + **字节水位**（距 125MB 上限留 1MB 余量）双保险 |

### 6. 12bpp 颜色偏蓝修复（追加，浏览器实测反馈）

**现象**：模拟器主题/状态栏颜色正常，但多数游戏/软件进入后整体偏蓝——白底变青色、
Adobe 红 logo 变绿色、黑色不变（Adobe Reader LE 启动画面实锤）。

**根因**：这三个颜色变换唯一吻合的解释是**通道整体错位一格**。Symbian 12bpp
（color4k）内存布局是 XRGB4444；桌面端按 GL RGBA4444 上传后用纹理 swizzle
`{G,B,A,one}` 重映射回正确通道，而 WebGL2 没有 TEXTURE_SWIZZLE，WASM 分支
直接跳过 → 采样到的是 (X,R,G,B)：白 (X,F,F,F)→青 (0,255,255)、红 (X,F,0,0)→绿。
主题用的是 16bpp/32bpp 位图（565 本就是直序、32bpp 已有 CPU 侧 R/B 交换）所以正常。

**修复**（`graphics_driver_shared.cpp` `update_bitmap` WASM 分支）：12bpp 数据上传
前 CPU 侧每像素 `out = (in << 4) | 0xF`——XRGB 旋转成 RGBA 并强制不透明（对应
桌面 swizzle 的 alpha=one）。所有 12bpp 上传（bitmap_cache / gstore / DSA）都
汇聚到这一个函数，一处修复全覆盖；data 指针总是生产者新分配的堆拷贝，原地改安全。

### 7. 触摸输入修复 + SIS 安装即时刷新（追加，浏览器实测反馈）

**触摸屏完全无效的两个根因**（`src/emu/web/src/main.cpp` `sdl_web_window`）：

1. **按键编码错位**：`find_most_significant_bit_one()` 返回 1-based 位号，SDL 左键
   （`SDL_BUTTON_LEFT=1`，索引非掩码）被算成 `1` = `mouse_button_right` → winserv
   `make_mouse_event` 把每次点击都派发成 **button3down**（右键），而触摸应用只认
   button1down。修复：显式映射 SDL 索引→driver 枚举（左=0/右=1/中=2），MOTION 用
   状态掩码判断。坐标无需修——emscripten-ports 的 SDL2 鼠标事件自带 CSS 缩放补偿，
   finger 事件是归一化坐标。
2. **FINGERUP 坐标为 (0,0)**：按下在控件上、抬起在左上角，UI 一律判定"取消点击"。
   修复：释放事件携带真实抬起坐标。

附带：触屏设备上 SDL 会同时合成鼠标事件（`which == SDL_TOUCH_MOUSEID`）造成双投递，
已过滤；finger id（浏览器 touch identifier 无界递增）映射到小槽位（复用
`active_pointers_`，Symbian 指针号须 < MAX_SYMBIAN_SUPPORTED_POINTERS）。

**SIS 安装后列表不刷新**：applist 服务只在启动/换盘时扫描注册表，桌面 Qt 装完会调
`force_refresh_applist()` → `rescan_registries()`，web 端漏了这步 →
`wasm_install_package` 成功后同步调用 `alserv->rescan_registries(io)`（WASM 下
rescan 本就是同步路径），JS 端 `refreshApps()` 立即可见新应用。

**搜索框输入不了英文**：Emscripten SDL2 默认把键盘监听挂在整个 window 并对每个
keydown `preventDefault()`——页面 HTML 输入框的裸按键全被吞掉（中文反而能进，
因为 IME 走 composition 事件不受影响）。修复：`SDL_SetHint(
SDL_HINT_EMSCRIPTEN_KEYBOARD_ELEMENT, "#canvas")`，键盘只在 canvas 获得焦点时
被模拟器捕获；运行页本就会在游戏开始/点击画面时 `canvas.focus()`，虚拟键盘
pointerdown 有 preventDefault 不抢焦点，物理键盘游戏输入不受影响。

### 8. 5320 计算器按 LSK 卡死——排查与三项修复（追加）

**症状**：触摸 ROM（5320）计算器进入后按左软键（Options）卡住，控制台零输出。

零输出 + 卡住的可能形态：① guest 线程死等（模拟器活着）② 主线程 C++ 死循环
③ 跨线程互锁（页面全冻）。无法本地复现（ROM 在用户浏览器 IndexedDB），本轮
修了三个排查中实锤的 bug + 配了诊断工具：

1. **MMF 音频回调死等（最高嫌疑）**：no-op dsp 流的 write() 从不触发
   `more_buffer` 回调，而 mmf dev server 靠它完成 guest 的 BufferToBeFilled
   请求——任何走 DevSound 的播放（按键音/提示音）第一块缓冲后状态机永久卡住，
   等它的 guest 线程死等；后续对该线程的**同步** IPC 全部跟着挂。修复：no-op 流
   加 20ms 节拍线程（detached + shared_ptr 状态块，不 join——WASM 主线程禁
   Atomics.wait），按写入节奏消费缓冲并在节拍线程上触发回调（等价真实声卡的
   音频线程模型）。
2. **keysound 空指针**：HLE keysound 服务 `aud_out_ = new_output_stream(...)`
   在 null 音频驱动下返回 nullptr，`play_sid` 直接 `aud_out_->is_playing()`
   解引用。加判空跳过（无声不崩）。
3. **dirty_tlb_page 跨线程竞争**（上一轮引入）：timer 线程 unmap 代码页会直接
   清 instruction_cache，与主线程解释器并发读写——unordered_map 桶链被破坏后
   find() 可能**死循环**（卡死+零输出的完美样本）。修复：跨线程只置
   `icache_invalidate_pending` 原子标志，DISPATCH 在块边界消费（该处重置翻译
   缓冲安全）。

**诊断工具**：新导出 `wasm_debug_dump()`（boot.js 封装 `EKA2L1.debugDump()`）。
卡住时在 DevTools 控制台调两次（间隔几秒）对比：
- `guest_instrs` 不动 → guest 全阻塞，看各线程 `state`/`wait_obj` 是谁在等什么
- `guest_instrs` 在涨且 PC 不变 → guest 在该 PC 死循环（dump 给出 模块名+偏移）
- `debugDump()` 本身无返回/页面全死 → 主线程卡在 wasm 内部（另一类 bug，直接反馈）
输出含：每线程 state/PC(模块+偏移)/LR/SP/wait_obj、当前线程标记、dyncom 指令与
翻译块计数、redraw 计数、paused 标志。

### 9. 5320 计算器按 LSK"卡死"——实为 guest panic，已定位并修复

上一轮三项修复后用户复测仍卡。本轮用 `EKA2L1.debugDump()` 抓现场：dump 里
**只剩 4 个系统服务线程、计算器进程消失**——**不是卡死，是 app panic 退出了**，
画面停在最后一帧；控制台零输出是因为 panic 日志原本是 `LOG_TRACE`（web 默认
warn 级被过滤）。

**定位手段**（全部本轮新增，已入库）：
1. panic 日志 `LOG_TRACE` → `LOG_WARN`（[thread.cpp](src/emu/kernel/src/thread.cpp)），
   app 退出日志同样提级（main.cpp）——以后 app 静默死会直接打出来
2. `thread::dump_panic_context()`：panic 时打印 PC/LR（解析到"模块+偏移"）、SP、
   启发式栈回溯（扫栈找落在某 codeseg text 段的字），外加每个返回地址周边代码
   hex（配合本机 python capstone 反汇编 stripped ROM）
3. 本地无头复现 harness：`build_wasm_test/bin/dev-test.html`（装 ROM→启动计算器
   →注入 LSK→两次 dump），headless Chrome 驱动、设备持久化到测试 profile 加速迭代
   （**不入库**：引用本地 ROM 路径）

**根因链**（反汇编 eikcoctl/calcsoft 得到）：
- panic = `EIKCOCTL 8` = `CEikMenuPane::DeleteMenuItem(cmdId)` 找不到该命令项
- 调用者 = 计算器 `DynInitMenuPaneL`，逻辑：
  `if (FeatureManager::FeatureSupported(1012)) return; else { DeleteMenuItem(base+0x11); DeleteMenuItem(base+0xF); }`
- 探针证实：LSK 后查询 `FeatureSupported(1012) → 0`（不支持）→ 走删除分支 →
  第二项 `base+0xF` 不在 pane → panic
- 再查 `featreg.cfg`：**整个文件只有 24 字节**（`"feat"` + ver=0 + entry数=0 +
  range数=1 + 1 个厂商 UID 范围 `[0x10279806..0x10281805]`），标准小号特性一律
  default-deny → EKA2L1 对 1012 返回 0。真机计算器正常 = 真机视 1012 为支持

**修复**（[featmgr.cpp](src/emu/services/src/featmgr/featmgr.cpp)）：`do_feature_scanning`
里把特性 `1012` 加入已支持列表（`feature_id_app_menu_show_images`）。验证：harness
重跑，0 panic，LSK 后计算器线程仍存活（`state=wait_fast_sema`，菜单已弹出等输入），
redraws 继续增长。

> 注：完整修复应实现 Symbian"默认支持"特性语义（多数 ROM 的 featreg.cfg 近乎空），
> 目前是针对性补单个特性，爆炸半径仅一个 id（不会误开日语 1080 等）。其他 app 若
> 因别的特性 default-deny 而崩，同法定位（debugDump + 反汇编）后按需补。

### 10. 七夜（7 Days - Salvation）启动失败——已深度诊断，未完全修复

`.sisx` 装好、能列出、点击启动后进程跑约 5700 万条指令做了不少 UI 初始化
（窗口组、皮肤、redraw、OOM AKNCAP）然后**启动失败**。两种表现（同一 init
失败的两面）：
- 用户实机：panic `E32USER-CBase 40` = `~CActive` 销毁时仍有未完成请求
  （ConstructL 中途 LEAVE → TRAP cleanup 析构半成品 active object，euser 0x174xx
  就是 cleanup stack）
- 无头 harness 复现：`User::Exit(0)` 干净退出（exit_type=0）

诊断手段（本轮新增 Leave 日志 + 复用 panic dump）：
- `leave_start` 打印 leave code+线程（TRACE 级，调试时降日志级别看）
- 反汇编用户给的栈：euser `0x801a4e88` 确认是 E32USER-CBase panic 函数；游戏
  0x9B950 是 ConstructL 在建带子对象的对象，中途 leave

观察到的 leave 序列（7Days 线程 init 期）：`KErrNotFound(-1)` ×3 →
`KErrAlreadyExists(-5)` → 退出。多数 leave 被 TRAP 正常捕获，致命的是某个未捕获的。

观察到的 emulator 缺口（均**非即时致命**，游戏越过它们后才退）：
- 未实现 exec SWI `0x0A` 和 `0x82`（EKA2L1 v94 表的真实空缺；注意
  exception_handler 是 0x5A 不是 0xA，故 0xA 是别的 EExec 调用，需 Symbian
  `EExec` 慢执行枚举才能确认第 10/130 号是什么）
- 皮肤解析 `skn.cpp: Unknown class def chunk type: 12`
- `Can't find requested screen 134217728`（已回退到焦点屏，疑似无害）

**为何难**：不像计算器是单点特性门，七夜是多因素 init 失败。盲目实现猜测的
exec 调用会危及其他 app。完成它需要：①给 leave 加"前台 app + 深层/未捕获"
条件的栈回溯，抓到那个致命 leave 的应用侧调用方；②用 Symbian EExec 枚举确认
并实现 0x0A/0x82；③确认游戏数据文件是否装在它查找的路径。

#### 10b. 第 8 轮续查（2026-06-10 深夜）：确认并实现缺失的 EKA2 LDD exec

复用无头 harness（`build_wasm_test/bin/dev-test.html`，本地引用 ROM，不入库），
给 `leave_start` / `thread_kill` / `process_kill` / 未实现 SVC 加上**可由前端开关

ROM 线性映射在 0x80000000）+ 运行期探针三方交叉确认，**0x0A / 0x82 不是猜的**：
- `0x0A = ChannelRequest`（`RBusLogicalChannel::DoControl/DoRequest` 统一入口，
  按 func 符号分派；EKA2L1 已有 `logical_channel_do_control/do_request`，只是 v93/v94

- `0x81(v93) / 0x82(v94) = DeviceLoad`（`User::LoadLogicalDevice`，**8-bit** 文件名 +

注：v93 比 v94 整体 -1（缺 `ServerCreateWithOptions`），故同一调用在两表号差 1；
v93 号已被 ROM 实测验证（探针打出正确设备名 `displaydriver`），v94 号按 +1 平移推得。

### 3. profiling 火焰图分析（带符号版，主线程 ~12s 窗口）

|---|---|---|

| `(idle)` | 11.2% | 空闲 |

### 4. 已修：glGetError 每帧海量调用（6.6% 主线程，零风险）

**根因**：`init_gl_graphics_library()` 把错误检查函数 `gl_post_callback_for_error`
注册成了 **glad 全局 post-callback**——**每调用一次 GL 函数后都跑一遍 `glGetError()`**。
WebGL 里 `glGetError` 强制同步往返/flush，每帧成千上万次 GL 调用全被它拖累。它只是
调试用的错误日志，对功能无影响。

**修复**（`graphics_ogl.cpp`）：WASM 下不注册该 post-callback。

```cpp
#if !EKA2L1_PLATFORM(WASM)
        glad_set_post_callback(gl_post_callback_for_error);
#else
        (void)&gl_post_callback_for_error;
#endif
```

### 1. 复核发现 ①：第 9 轮 glGetError"修复"根本无效（已重修）

**实测**：基线跑 30s，`ctx.getError` 调用 **~634 次/秒持续增长**——第 9 轮"WASM 下
不注册 glad post-callback"没有任何效果。

**根因**：仓库的 glad 是 **c-debug 变体**，每个 GL 函数都经过 debug 包装层，包装层
没注册自定义回调时会调用**默认回调** `_post_call_callback_default_gl`（glad.c:38），
而它本身就执行 `glad_glGetError()`。所以"不注册"恰恰留下了每调用一次 GL 就同步
glGetError 一次的默认行为。

**正确修复**（`graphics_ogl.cpp`）：注册**空操作回调**顶掉默认回调：
```cpp
#if !EKA2L1_PLATFORM(WASM)
        glad_set_post_callback(gl_post_callback_for_error);
#else
        glad_set_post_callback(gl_post_callback_no_op);   // 顶掉 glad 默认的 glGetError
#endif
```

**验证**：修后 30s 内 getError 恒定 **55 次**（全部来自初始化），稳态归零。

顺手修掉同类问题：`set_line_width` 每次执行都 `glGetIntegerv(GL_ALIASED_LINE_WIDTH_RANGE)`
+ `glGetError()`（GLES 游戏每帧 init_context_state 都发该命令）→ 改为首次查询后缓存
range、本地校验，不再每次问驱动。

### 3. dyncom 优化 ①：TLB 快路径内联进 armstate.h

`ReadMemory32`/`WriteMemory32` 等 8 个访存函数此前在 armstate.cpp 出线，解释器每次
访存都付一次函数调用开销（profile 中占 9.3% self）。改法：

- `ARMul_State` 新增 `r12l1::tlb *mem_cache_direct` 成员（构造时取
  `core->mem_cache()`，地址终生稳定），armstate.h 直接 include `cpu/12l1r/tlb.h`。
- 8 个函数改为 header 内联：TLB 命中 → 直接读写宿主指针返回；未命中或大端模式 →
  调出线的 `*Slow()`（保留原完整逻辑，语义零变化）。
- 大端语义保持与原版逐字节一致（命中路径从不 swap，原版即如此；Symbian 全程小端）。

### 4. dyncom 优化 ②：shtop/get_addr 去虚化（函数指针 → 索引 + switch）

译码后的指令结构体原来存两类**函数指针**，解释器每条 ALU/访存指令都要做一次
**间接调用**（WASM `call_indirect` 还带表边界 + 签名检查，比原生更贵）：

- `shtop_func`（11 种 shifter operand 求值器）——每条带 operand2 的数据处理指令一次
- `get_addr`（20 种寻址模式）——每条 LDR/STR/LDM/STM 一次

改法（`arm_dyncom_trans.h` + `arm_dyncom_interpreter.cpp` + `arm_dyncom_trans.cpp`）：

- 结构体字段 `shtop_fp_t shtop_func` → `unsigned shtop_idx`；`get_addr_fp_t get_addr`
  → `unsigned addr_mode`（两个新枚举 `shtop_index`/`addr_mode_index`）。
- `GetShifterOp`/`GetAddressingOp`/`GetAddressingOpLoadStoreT` 改为返回索引（译码
  逻辑一行未动）。
- 解释器新增 `EvalShifterOperand`/`EvalGetAddr`：对索引 switch，编译器把 31 个小函数
  体直接内联进跳转表，间接调用变直接跳转。19 处 `inst_cream->get_addr(...)` 调用点
  与 `SHIFTER_OPERAND` 宏统一替换。
- 顺手：4 个 `LdnStM` 寻址函数里逐位 while 循环数寄存器 → `__builtin_popcount`
  （wasm 是单条 `i32.popcnt`；MSVC 走 SWAR 兜底）。LDM/STM 是函数序言/尾声高频指令。

## 十四、第 11 轮（2026-06-11）：iOS 装 ROM"立即刷新"二修（内存基线 −130MB+ + 延迟启动）

用户反馈：iOS 装 ROM 仍然"立即刷新，导致安装不了"。上一轮（48f7b08e5）已做
256MB/分块上传/先存后刷新，没治本。本轮从"常驻基线"和"被杀时机"两头下手。

### 1. 重新定位：刷新发生在哪一刻

"立即刷新" = iOS Safari 把整个 WebContent 进程 jetsam 后自动重载页面。最脆弱的时间
窗不是安装中，而是**打开「文件」选择器的瞬间**：document picker 把页面挤到后台，
此时页面顶着几百 MB 常驻内存，正是 jetsam 的首选目标——被杀后回到 Safari 就是
"立即刷新"，选好的文件也丢了，根本到不了安装那一步。

### 2. 常驻内存大头盘点（本轮量化出来的）

| 项 | 占用 | 问题 |
|---|---|---|
| dyncom `TRANS_CACHE_SIZE` | **131MB** | `ARMul_State` 的内联成员数组，创建核心时一次性 malloc 131MB；实测 30s 重度运行只用到几 MB～几十 MB |
| `INITIAL_MEMORY` | 256MB | 为 131MB 翻译缓存留的余量；iOS 对共享 wasm 内存的提交策略保守，整块计入配额 |

| `PTHREAD_POOL_SIZE=16` | 16 个预热 worker | 实际稳态线程 ≈3 个（ntimer + applist 残留 + spdlog） |

### 3. 修复

| # | 改动 | 文件 |

| 1 | `TRANS_CACHE_SIZE`：WASM 131MB → **32MB**（溢出有现成安全网：到 `TRANS_CACHE_FLUSH_MARGIN` 自动 flush 重来，只是变慢不会崩；桌面保持原值） | `armstate.h` |

## 十五、第 12 轮（2026-06-11）：iOS 装 ROM 三修——流式 RPKG 安装（峰值再砍一个包大小）

第 11 轮（基线 −130MB + 延迟启动）后用户反馈 iOS 仍崩。安装期的瞬时峰值还有一个
结构性大头没动：**整个 RPKG 必须先复制进 MEMFS**（= JS 堆）才能交给
`install_rpkg()`，5320 包 115MB，别的机型包几百 MB——这一份临时副本叠在解压产物
（~130MB+）上，依旧能把 iOS 顶过线。

### 1. 关键洞察：install_rpkg 是严格顺序读

读了 [rpkg.cpp](src/emu/system/src/installation/rpkg.cpp)：开头一次 `fseek(END)`
只为进度条总量，之后全程顺序 `fread`（头 → 逐条 entry 元数据/路径 → 数据按 64KB
边读边写出），没有任何回跳。**顺序读 = 可以改成推式（push）流**。

### 2. 实现：三层打通

| 层 | 内容 |
|---|---|
| C++ `rpkg.cpp/h` | ① 把 install_rpkg 安装后半段（识别产品/注册设备/temp 改名）提取为 `finalize_extracted_rpkg()` 共用；② 新增 `rpkg_stream_installer` 类：状态机逐字节复刻原解析（16B magic 块 / v1=24B v2=32B 头校验 / entry 元数据 24B / UTF-16 路径+data_size / 数据直写输出流），含原版没有的 path_len 健壮性护栏（<4 或 >0x8000 判损坏），截断包行为与原版一致（停止解析仍 finalize） |
| `web/main.cpp` | 5 个导出：`wasm_rpkg_stream_begin/buffer/feed/finish/abort` + `rom_target`。begin 拒绝已有设备（-1005 复用现有错误解码）；finish 注册设备 + `save_devices()` + 算出 `roms/<firmcode>/SYM.ROM` 目标路径 |
| JS | `boot.js` 新增 `EKA2L1.streamInstallRpkg(file)`：8MB 切片 → `HEAPU8.set` 进 wasm 内部缓冲 → feed；`index.js` 安装流程：**RPKG 流式安装（不落 MEMFS）→ ROM 直写最终位置（不再有临时副本+拷贝）→ `initDevice('','')` 激活**。EKA1 纯 ROM 与流式不可用时回退旧路径。`EXPORTED_RUNTIME_METHODS` 加 `HEAPU8` |

安装期内存账（5320 为例）：旧 = 解压产物 ~130MB + RPKG 副本 115MB + ROM 临时副本；
新 = 解压产物 ~130MB + 8MB 滑动块。**峰值砍掉整个包的大小**，包越大省得越多。

### 3. 静默死亡面包屑（iPhone 无控制台的诊断闭环）

iOS jetsam 杀页**没有任何错误事件**，用户只看到"刷新了"。现在 `index.js` 在安装各
阶段写 `localStorage('eka2l1_install_stage')`（启动核心/流式安装 RPKG x%/写入 ROM/
激活设备/保存到浏览器），成功或已上屏的失败都会清除；**下次加载若发现残留标记，
红框直接告诉用户"上次安装在「X」阶段中断"**+ 应对建议。下一轮迭代不再盲猜。

### 4. 回归（headless Chrome 全新 profile）

- 流式安装：27%→55%→83% 进度 → `rom target: /eka2l1/roms/rm-409/SYM.ROM` →
  `initDevice('','') → 0` ✅（固件识别/设备注册/devices.yml 全对）
- 天地道照常运行（redraw/吞吐与上轮持平）✅
- 旧路径保留：流式不可用时自动回退 ✅

### 仍可能的残余风险（按面包屑反馈再迭代）

- 「保存到浏览器」阶段：IDBFS syncfs 把全部解压内容塞进**单个 IndexedDB 事务**，
  结构化克隆有一份 ~130MB 量级的瞬时副本。若用户反馈中断集中在此阶段，下一步做

## 十七、第 14 轮（2026-06-11）：ARM→wasm 运行时 JIT 落地（redraw 再 +51%，累计 2.4×）+ iOS 无痕模式定案

### 1. iOS「保存到浏览器」中断定案：无痕模式

用户 iPhone 17 Pro（12GB）仍中断于「保存到浏览器」——面包屑首次回传成功。两个发现：
① 部署在 eka2l1.zixing.fun 的是**旧构建**（第 13 轮的分批保存没上线）；② 截图地址栏
是**无痕模式**（面具图标）——iOS 无痕的 IndexedDB 在**内存**里，保存=把几百 MB 再
复制进 RAM，12GB 设备照样被 jetsam，且根本不会持久化。
**修复**：`installDevice` 前置 `navigator.storage.estimate()` 检查，配额 <300MB
（无痕特征）直接红框拦截并说明换普通窗口。

### 2. JIT：浏览器版的"大招"（v86 同款方案）

浏览器不能执行原生机器码，但能**即时编译 WebAssembly**：热翻译块→生成迷你 wasm
模块（导入模拟器自身的共享内存）→ 同步 `new WebAssembly.Module/Instance` →
`wasmTable.grow+set` 放进共享间接函数表 → 解释器 DISPATCH 此后按**普通函数指针**
调用。冷块/不支持的块留在解释器——**正确性永不依赖覆盖率**。

| 组件 | 内容 |
|---|---|
| `arm_dyncom_wasmjit.cpp`（新） | 手写 wasm 字节码发射器（LEB128/段封装）+ 块编译器。NZCV 放 wasm 局部变量（进出块时与解释器的 dummy 标志字段同步）；访存走**同一个 r12l1 TLB**（内联查找，miss/权限不符→bail）；任何 bail 都把 `Reg[15]` 设到第一条未执行指令并返回已执行数，解释器无缝续跑 |
| 指令子集 v1 | ALU 全家（AND…MVN，S 标志完整，含 AddWithCarry 的 C/V）、CMP/CMN/TST/TEQ、操作数 imm/reg/LSL/LSR/ASR-imm、LDR/STR/LDRB/STRB（立即数偏移）、BBL/BX/B_2_THUMB/B_COND_THUMB、ALU 写 PC（S=0）做块终结。其余→提前断块（前缀 ≥4 条才编译） |
| 编译源 | **翻译后的 arm_inst 流**（解释器同款），Thumb 已被翻译器归一化，免重复解码；`arm_instruction_trans[]` 表序在 trans.cpp 静态自检导出（重排不脱钩） |
| 触发 | block_lookup 热度计数 ≥128 → 编译；`invalidate_translation_cache()` 同步释放表槽（空闲表复用） |
| 开关 | 默认开；`?jit=0` 关（全局开关实时读取——CPU 核心在 boot 即创建，构造期锁存会失效，踩过）；`?jitlimit=N` 二分定位用 |

### 3. 调试战记：二分 + 离线差分对拍（方法论比 bug 本身值钱）

首跑游戏**卡死**（guest 指令数冻结）。流程：
1. `?jitlimit=N` 二分（0✓ 8✓ 16✓ 21✓ **22✗**）→ 坏块=第 22 个；
2. printf 块信息：pc=0x80191A54、ARM、20 条 `[MOV, 5×(RSB,SUB,ADC), BBL]`——
   一眼 `__aeabi_uidiv` 除法内环（进位链重灾区）；本地 capstone 反汇编 ROM 证实；
3. 该块**纯寄存器运算**→把发射的模块字节 hex dump 出来，**node 离线实例化**，与按
   反汇编手写的 JS 参考实现对拍 512 组随机输入→复现错位；
4. 自写 wasm 反汇编器打印发射码→**bug 现形**：`emit_awc` 把操作数归一化进 T0/T1，
   而 RSB/RSC 以 (lhs=T1, rhs=T0) 调用，第一步 `local.set T0` 先覆盖了还没读的
   Rn。改为先读两操作数进独立暂存（T2/ADDR）再写。修后 512 组全过、游戏正常。
5. 顺手修：TLB 空条目 host_base=0 时 guest 0 页访问会假命中读穿——加 host_base
   非零检查（原 lookup() 有此守卫，发射版漏了）。

### 4b. 红框 v2 回传定案 + 保存先于激活（关键重排）

部署后 iPhone 红框给出完整现场：「保存:共 9378 项 181MB」+ ROM 66MB/RPKG 115MB/
配额 38GB（非无痕）——**死于扫描完成后、第一批 4MB 写入完成前**。醒悟：流式安装
路径里「激活设备」（= 完整模拟器启动，载入 66MB ROM、wasm 堆涨数百 MB）发生在
保存**之前**，保存是在内存峰值上跑的。**重排：流式装机 → 写 ROM → 分批保存
（内存地板）→ 打持久化标记 → 激活**。从此哪怕激活阶段被杀，刷新页面直接从
IndexedDB 恢复设备——自愈闭环。回归：新顺序装机+保存+激活+游戏 ✓，重载恢复
`initDevice('')→0` ✓。

### 5b. JIT v2（第 15 轮）：仪表驱动扩指令集，覆盖率 48%→85%

iOS 通了之后继续压性能。先加**两块仪表**：① jit 指令占比（jit_instrs/总数）；
② 拦路虎直方图（每次编译失败记录第一条不支持指令的 idx）——优化照单抓药：

| 拦路虎（按频次） | 处置 |
|---|---|
| CPY（thumb `mov rd,rm` 别名，榜首） | = 无标志 MOV，cream 同 mov_inst，直接复用 |
| LDM/STM 全家（5 个表项） | plain 形式编译：**先解析首尾两页宿主基址→写回→逐字经已捕获基址访问**（bail 幂等 + 免疫定时器线程并发 TLB 失效）；STM 的 old_RN、PC+8 怪癖逐一镜像；LDM 带 PC 终结块（TFlag=bit0） |
| LDR/STR/LDRB/STRB 变址 | reg-offset、imm/reg pre/post 写回：先解析后写回，宿主指针捕获后访问 |
| LDRH/STRH/LDRSH/LDRSB | MLnS imm/reg 寻址 + wasm load16_u/s、load8_s、store16 |
| MUL/MLA、thumb BL 调用对 | 直译 |

又一次二分战记：v2 首跑卡死 → `?jitlimit` 二分到 #13（thumb `push {r4,lr}`）→
**原始 Rn 暂存在 T2，而每次 TLB 解析内部也用 T2 当暂存** → 写回时 SP=TLB条目指针-8。
教训：发射器的暂存寄存器分配要有清晰的 clobber 约定（新增 L_T3 专用于跨解析存活）。

**结果**：JIT 指令占比 47.9%→**83-88%**；天地道 redraw 12.1→**13.3/s**（较第 9 轮
基线累计 **2.7×**）；七夜 +16%。剩余拦路虎为低频长尾（scaled-reg 寻址、BLX、
SMULL、MSR、SWI）。下一档：块直链/循环内联（消 DISPATCH 往返）、scaled-reg 寻址。
